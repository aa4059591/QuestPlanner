## 1. Business purpose

**1.1 Project background & motivation**

현대 사회를 살아가는 사람들에게 '계획'은 선택이 아닌 일상을 지탱하는 필수가 되었다. 끝없이 쏟아지는 일상 속 할 일들과 자기계발을 위한 목표들을 잊지 않고 체계적으로 관리하기 위해, 많은 사람들이 투두리스트 앱을 적극적으로 활용하고 있다. 특히 학생 신분인 나에게 플래너는 성공적인 학업과 일상 관리를 위해 필수다. 매 학기 여러 수업을 들으며 과제와 프로젝트, 시험을 위해 준비해야할 공부는 물론 자기계발을 위한 개인적인 어학 공부나 자격증 등 여러 목표를 머릿속에만 담아두기에는 벅차다. 인간의 기억력에는 한계가 있기에 기록하지 않으면 중요한 일정을 잊어버리기 쉽고, 이는 곧바로 학업 성취도 저하와 목표가 무너지는 치명적인 결과로 이어진다. 그렇기에 나는 오래전부터 해야 할 일을 기록하고 체크하는 투두리스트의 필요성을 느끼며 시중에 출시된 다양한 앱들을 사용해 왔다. 하지만 초반에만 열정적으로 기록해가며 실행하지만 오래가지못하고, 기록만 해두며 완료하지 못하여 방치하게 되는 경험이 많았다. 

기존의 투두리스트 앱들이 가진 근본적인 한계는 바로 '동기 부여의 부재'와 '지속성의 결여'에 있다. 대부분의 투두리스트 앱들은 사용자가 할 일을 텍스트로 입력하고, 완료 시 체크박스를 클릭하며 취소선을 긋는 단조롭고 획일적인 기능만을 제공한다. 물론 자신이 계획한 일들을 하나씩 지워나가는 과정에서 시각적인 진행 상황을 확인하고, 완료된 일들을 보며 나름의 성취감을 얻을 수는 있지만 이러한 성취감은 오로지 사용자 스스로의 강한 인내심과 굳건한 내적 동기에만 의존해야 한다는 약점이 있다. 투두리스트를 꾸준히 작성하고 실천하여 진정한 자기효능감을 맛보기 위해서는 상당한 시간이 필요하지만 그 단계에 도달하기도 전에 매일 같은 행위를 반복하는 자체가 또 하나의 귀찮은 과제이자 피로감으로 다가오게 된다. 나 역시 매번 성취감을 제대로 느끼기도 전에 단순히 기록하는 과정의 지루함을 이기지 못하고 중도 하차하기 일쑤였다. 즉, 나처럼 느끼는 사용자들에게 진정으로 필요한 것은 단순한 일정 기록장이 아니라 그 기록을 강제적이지 않으면서도 즐겁게 지속하게 만들어줄 강력하고 즉각적인 외적 보상 시스템이라는 결론에 도달하게 되었다.

이러한 기존 앱들의 명확한 한계를 극복하고자, 나는 사람들이 가장 쉽게 몰입하고 열광하는 매체인 '게임'의 요소를 투두리스트와 결합하는 '게이미피케이션(Gamification)'을 기획하게 되었다. 게임이 사람들을 끊임없이 몰입하게 만드는 핵심은 능동적인 참여에 대한 즉각적인 피드백과 성장 및 수집 욕구의 충족에 있다. 나는 이 심리적 장치를 현실의 목표 달성 과정에 도입하여, 사용자가 플래너에 적힌 할 일을 완료하는 행위 자체가 긍정적인 기대감을 유발하는 매개체가 되도록 기획했다. 현실의 지루하고 고단한 과제 수행을 마치 게임 속 퀘스트를 달성하는 과정으로 탈바꿈시킴으로써, 텍스트를 지우는 단순한 행위를 넘어 시각적인 보상과 수집의 재미를 제공하고자 한 것이다. 이를 통해 사용자는 억지로 인내심을 짜내는 것이 아니라, 스스로 호기심과 성취욕에 이끌려 목표를 완수하게 되는 자발적인 선순환을 경험하게 될 것이라 판단했다.

더 나아가 이 기획은 인간의 보편적인 심리인 '손실 회피 성향'을 긍정적인 오기와 실천력으로 승화시키는 데 초점을 맞추었다. 며칠에 걸쳐 현실의 할 일을 성실히 수행하며 알을 부화시키고 캐릭터를 진화시켜 나가는 과정에서, 사용자는 자신의 캐릭터에 깊은 애착을 형성하게 된다. 오늘 하루 나태해져서 계획을 미루게 되면, 내 캐릭터의 성장이 지연된다는 심리적 요인이 작용해 지금까지 키워온 캐릭터와 노력이 아까워서라도 오늘 완료하고 싶다는 강렬한 오기를 발동시키는 훌륭한 촉매제가 될 것이라 확신했다. 또한, 획일화된 기준을 강요하기보다 사용자 스스로 목표의 비중을 설정하고 자신만의 속도에 맞춰 성취감을 조절할 수 있는 유연성을 부여해 장기적인 지속 가능성을 확보하고자 했다. 

마지막으로, 혼자서만 계획을 실천할 때 의지력이 고갈되어 스스로와 쉽게 타협하게 되는 문제를 해결하기 위해 '길드'라는 소셜 네트워크 시스템을 도입하기로 구상했다. 혼자만이 아니라 친구, 스터디원, 혹은 비슷한 목표를 가진 타인과 그룹을 맺고 서로의 진행률과 상태를 실시간으로 공유하는 환경을 만드는 것이다. 다른 길드원들이 매일 성실하게 목표를 달성하며 레벨을 올려나가는 모습을 지켜보는 것은 사용자에게 엄청난 시너지 효과와 건강한 경쟁의식을 불러일으킨다. 서로의 성취를 자극제로 삼고 퀘스트를 수행하는 듯한 연대감을 부여함으로써, 더 이상 외롭고 지루한 자기관리의 늪에 빠지지 않게 방지하는 것이. 결과적으로 본 프로젝트는 '일정 관리'라는 본연의 목적을 넘어, 일상의 고단한 과업을 게임처럼 즐겁게 수행하고 타인과 함께 성장해 나가도록 이끄는 동기부여 플랫폼을 만들고자 하는 필요성에서 출발하게 되었다.


**1.2 Goal**

본 프로젝트의 최종 목표는 사용자의 일정 관리 효율성을 높임과 동시에, 목표 달성 과정의 지루함을 없애주는 'RPG 기반의 할 일 관리 클라이언트 시스템'과 타인의 상태를 실시간으로 공유하고 경쟁할 수 있는 '길드 서버 시스템'을 구축하는 것이다. 단순한 기록 도구를 넘어 사용자가 캐릭터를 육성하는 재미를 통해 매일의 투두리스트를 자발적으로 완수하게 만드는 것이 최종 목표이다.


**1.3 Target market**
- 매일 반복되는 일상과 목표 달성에 지루함을 느끼고 새로운 동기부여가 필요한 학생 및 직장인
- 게임적 요소를 좋아하는 사용자
- 자기 주도적 성취를 원하는 사용자
- 친구나 스터디원들과 함께 자극받으며 시너지 효과를 얻고 싶은 사용자

## 3. Use case list
**- Find use cases in your project.**

| Use Case | Actor | Description |
| :--- | :--- | :--- |
| **1) Sign Up** | User | 새로운 닉네임을 입력하여 고유한 계정 데이터를 생성한다. |
| **2) Login** | User | 기존에 생성한 닉네임과 데이터를 불러와 시스템에 접속한다. |
| **3) Save/Load Data** | System | 로컬 JSON 파일을 이용해 사용자의 경험치, 도감, 퀘스트 내역을 저장하고 불러온다. |
| **4) Add Quest** | User | 목표 날짜와 할 일의 내용을 작성하여 새로운 퀘스트를 등록한다. |
| **5) Delete Quest** | User | 잘못 입력했거나 포기한 퀘스트를 목록에서 완전히 삭제한다. |
| **6) Set Custom Points** | User | 퀘스트의 난이도나 중요도를 스스로 판단하여 보상 경험치 포인트를 직접 지정한다. |
| **7) Complete Quest** | User | 체크박스를 눌러 퀘스트를 완료하고, 설정한 포인트만큼 캐릭터 경험치를 획득한다. |
| **8) Cancel Quest** | User | 완료했던 퀘스트를 다시 체크 해제할 경우, 획득했던 경험치를 즉시 차감한다. |
| **9) Evolve Character** | System | 누적 경험치가 특정 구간을 넘으면 무작위 확률로 새로운 동물 형태로 진화시킨다. |
| **10) Check Encyclopedia** | User | 지금까지 수집하고 진화시켜 온 캐릭터들의 목록과 해금 상태를 도감에서 확인한다. |
| **11) Create Guild** | User | 방 이름과 비밀번호를 설정하여 새로운 소셜 길드를 개설하고 방장 권한을 얻는다. |
| **12) Join Guild** | User | 다른 유저가 만든 방 이름과 비밀번호를 입력하여 해당 길드원으로 접속한다. |
| **13) Real-time Sync** | Server | 길드원의 데이터 변동 시 방 안의 모든 유저 화면에 실시간으로 업데이트를 전송한다. |
| **14) Offline Roster** | Server | 유저가 접속을 종료해도 서버 내 방명록 데이터를 통해 마지막 상태를 계속 유지한다. |
| **15) Leave Guild** | User | 참여 중인 길드에서 빠져나와 개인 모드로 돌아간다. |
| **16) Delegate Host** | User (Host) | 방장이 나갈 때 남아있는 다른 유저를 선택해 방장 권한을 넘겨준다. |
| **17) Disband Guild** | User (Host) | 방장이 현재 방을 강제로 해산시키고 모든 길드원을 퇴장시킨다. |

## 4. Concept of operation
**- Describe how your system works.**

### 1) Login & Data Initialization
| Category | Description |
| :--- | :--- |
| **Purpose** | 사용자 인증 및 로컬 성장 데이터 복구 |
| **Approach** | 사용자가 닉네임을 입력하면, 시스템은 로컬 경로의 JSON 파일을 조회하여 해당 사용자의 경험치, 레벨, 퀘스트 목록을 메모리로 로드한다. |
| **Dynamics** | 앱 실행 직후 메인 화면 진입 시 발생 |
| **Goals** | 개별 사용자 식별 및 영구적인 데이터 연속성 보장 |

### 2) Quest Management with Custom Points
| Category | Description |
| :--- | :--- |
| **Purpose** | 사용자 주도적인 목표 설정 및 보상 체계 구축 |
| **Approach** | 사용자가 할 일을 입력할 때, 해당 퀘스트의 난이도에 따라 획득할 경험치(EXP) 수치를 직접 입력받아 JSON 데이터에 할당한다. |
| **Dynamics** | 새로운 퀘스트를 추가하거나 완료/취소 상태를 변경할 때 발생 |
| **Goals** | 자율적인 보상 설정을 통한 투두리스트의 게임화 실현 |

### 3) Character Evolution & Encyclopedia
| Category | Description |
| :--- | :--- |
| **Purpose** | 수집 욕구 자극 및 시각적 성장 피드백 제공 |
| **Approach** | 누적 경험치가 레벨업 임계치에 도달하면, 내부 난수 생성 로직을 통해 무작위 캐릭터로 진화시키고 해당 데이터를 도감 목록에 업데이트한다. |
| **Dynamics** | 퀘스트 완료 체크로 인해 레벨업 조건이 충족될 때 발생 |
| **Goals** | 지속적인 목표 달성을 위한 동기 부여 및 수집 콘텐츠 제공 |

### 4) Socket-based Guild Networking
| Category | Description |
| :--- | :--- |
| **Purpose** | 실시간 사용자 간 성장 상태 공유 및 소셜 상호작용 |
| **Approach** | 입력된 IP/Port 정보를 기반으로 서버와 소켓 연결을 수립하고, 상태 변화 발생 시 서버를 통해 같은 방의 모든 유저에게 데이터를 브로드캐스팅한다. |
| **Dynamics** | 길드 방 입장 후 퀘스트 완료, 레벨업 등 상태 변동이 발생할 때 실시간 작동 |
| **Goals** | 실시간 네트워크 동기화를 통한 소셜 경쟁 및 협력 환경 조성 |

### 5) Guild Authority & Session Management
| Category | Description |
| :--- | :--- |

| **Dynamics** | 방장이 길드 탈퇴를 시도하거나 방 해산 기능을 실행할 때 발생 |
| **Goals** | 유령 방 방지 및 안정적인 서버 세션 유지 관리 |

### 6) Local Data Synchronization
| Category | Description |

## 5. Problem statement
**- Describe the problems the project should be considered (including technical difficulties).**

**1) Problem #1: JSON 직렬화/역직렬화에 따른 데이터 하위 호환성 문제**
프로그램을 업데이트하여 캐릭터 환생 횟수 등 새로운 변수(데이터 구조)가 추가되었을 때, 기존에 저장된 로컬 JSON 파일을 불러오는 과정에서 `UnrecognizedPropertyException` 데이터 충돌 오류가 발생했다.
**해결:** Jackson 라이브러리의 `ObjectMapper` 설정에 `FAIL_ON_UNKNOWN_PROPERTIES`를 비활성화(false)하고 모델 클래스에 기본 생성자를 추가하여, 이전 버전의 파일도 유연하게 읽어올 수 있도록 하위 호환성을 보장하였다.

**2) Problem #2: 비동기 오프라인 동기화 (Zombie Room & Ghost User)**
클라이언트가 로그아웃하면 소켓 연결이 끊겨 다른 길드원의 화면에서 유저가 사라지거나, 방장이 방을 해산했는데도 오프라인 유저가 재접속하면 좀비 방이 부활하는 치명적인 네트워크 동기화 문제가 있었다.
**해결:** 서버(`SocialServer`) 내에 메모리 DB 역할인 `roomRosters`(방명록)와 `disbandedRooms`(해산된 방 블랙리스트)를 구현하였다. 접속이 끊겨도 방명록에 마지막 상태를 저장하여 길드원 목록을 유지하고, 폭파된 방에 재접속(RECONNECT) 시 즉시 클라이언트 룸을 삭제하도록 통제하였다.

**3) Non-Functional Requirements (NFRs)**
* **응답성:** 소켓 서버를 통한 데이터 브로드캐스팅(경험치, 레벨업 동기화)은 지연 없이 즉각적(1초 이내)으로 UI에 반영되어야 한다.
* **안정성:** UI의 화면 분할은 65(내 캐릭터) : 35(길드 현황)의 비율을 유지하며 해상도에 상관없이 쾌적한 가시성을 제공해야 한다.
* **보안성/독립성:** 유저의 개인 할 일(Quest) 데이터는 로컬 환경에만 독립적으로 보관되며, 네트워크 서버에는 오직 닉네임과 레벨, 경험치 퍼센트 정보만 공유되어야 한다.

## 6. Glossary

| Terms | Description |
| :--- | :--- |
| **QuestPlanner** | 본 프로젝트로 개발된 시스템의 고유 명칭. |
