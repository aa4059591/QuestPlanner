## 1. Business purpose

**1.1 Project background & motivation**

현대 사회를 살아가는 사람들에게 '계획'은 선택이 아닌 일상을 지탱하는 필수가 되었다. 끝없이 쏟아지는 일상 속 할 일들과 자기계발을 위한 목표들을 잊지 않고 체계적으로 관리하기 위해, 많은 사람들이 투두리스트 앱을 적극적으로 활용하고 있다. 특히 학생 신분인 나에게 플래너는 성공적인 학업과 일상 관리를 위해 필수다. 매 학기 여러 수업을 들으며 과제와 프로젝트, 시험을 위해 준비해야할 공부는 물론 자기계발을 위한 개인적인 어학 공부나 자격증 등 여러 목표를 머릿속에만 담아두기에는 벅차다. 인간의 기억력에는 한계가 있기에 기록하지 않으면 중요한 일정을 잊어버리기 쉽고, 이는 곧바로 학업 성취도 저하와 목표가 무너지는 치명적인 결과로 이어진다. 그렇기에 나는 오래전부터 해야 할 일을 기록하고 체크하는 투두리스트의 필요성을 느끼며 시중에 출시된 다양한 앱들을 사용해 왔다. 하지만 초반에만 열정적으로 기록해가며 실행하지만 오래가지못하고, 기록만 해두며 완료하지 못하여 방치하게 되는 경험이 많았다. 

기존의 투두리스트 앱들이 가진 근본적인 한계는 바로 '동기 부여의 부재'와 '지속성의 결여'에 있다. 대부분의 투두리스트 앱들은 사용자가 할 일을 텍스트로 입력하고, 완료 시 체크박스를 클릭하며 취소선을 긋는 단조롭고 획일적인 기능만을 제공한다. 물론 자신이 계획한 일들을 하나씩 지워나가는 과정에서 시각적인 진행 상황을 확인하고, 완료된 일들을 보며 나름의 성취감을 얻을 수는 있지만 이러한 성취감은 오로지 사용자 스스로의 강한 인내심과 굳건한 내적 동기에만 의존해야 한다는 약점이 있다. 투두리스트를 꾸준히 작성하고 실천하여 진정한 자기효능감을 맛보기 위해서는 상당한 시간이 필요하지만 그 단계에 도달하기도 전에 매일 같은 행위를 반복하는 자체가 또 하나의 귀찮은 과제이자 피로감으로 다가오게 된다. 나 역시 매번 성취감을 제대로 느끼기도 전에 단순히 기록하는 과정의 지루함을 이기지 못하고 중도 하차하기 일쑤였다. 즉, 나처럼 느끼는 사용자들에게 진정으로 필요한 것은 단순한 일정 기록장이 아니라 그 기록을 강제적이지 않으면서도 즐겁게 지속하게 만들어줄 강력하고 즉각적인 외적 보상 시스템이라는 결론에 도달하게 되었다.

이러한 기존 앱들의 명확한 한계를 극복하고자, 나는 사람들이 가장 쉽게 몰입하고 열광하는 매체인 '게임'의 요소를 투두리스트와 결합하는 '게이미피케이션(Gamification)'을 기획하게 되었다. 게임이 사람들을 끊임없이 몰입하게 만드는 핵심은 능동적인 참여에 대한 즉각적인 피드백과 성장 및 수집 욕구의 충족에 있다. 나는 이 심리적 장치를 현실의 목표 달성 과정에 도입하여, 사용자가 플래너에 적힌 할 일을 완료하는 행위 자체가 긍정적인 기대감을 유발하는 매개체가 되도록 기획했다. 현실의 지루하고 고단한 과제 수행을 마치 게임 속 퀘스트를 달성하는 과정으로 탈바꿈시킴으로써, 텍스트를 지우는 단순한 행위를 넘어 시각적인 보상과 수집의 재미를 제공하고자 한 것이다. 이를 통해 사용자는 억지로 인내심을 짜내는 것이 아니라, 스스로 호기심과 성취욕에 이끌려 목표를 완수하게 되는 자발적인 선순환을 경험하게 될 것이라 판단했다.

더 나아가 이 기획은 인간의 보편적인 심리인 '손실 회피 성향'을 긍정적인 오기와 실천력으로 승화시키는 데 초점을 맞추었다. 며칠에 걸쳐 현실의 할 일을 성실히 수행하며 알을 부화시키고 캐릭터를 진화시켜 나가는 과정에서, 사용자는 자신의 캐릭터에 깊은 애착을 형성하게 된다. 오늘 하루 나태해져서 계획을 미루게 되면, 내 캐릭터의 성장이 지연된다는 심리적 요인이 작용해 지금까지 키워온 캐릭터와 노력이 아까워서라도 오늘 완료하고 싶다는 강렬한 오기를 발동시키는 훌륭한 촉매제가 될 것이라 확신했다. 또한, 획일화된 기준을 강요하기보다 사용자 스스로 목표의 비중을 설정하고 자신만의 속도에 맞춰 성취감을 조절할 수 있는 유연성을 부여해 장기적인 지속 가능성을 확보하고자 했다. 

마지막으로, 혼자서만 계획을 실천할 때 의지력이 고갈되어 스스로와 쉽게 타협하게 되는 문제를 해결하기 위해 '길드'라는 소셜 네트워크 시스템을 도입하기로 구상했다. 혼자만이 아니라 친구, 스터디원, 혹은 비슷한 목표를 가진 타인과 그룹을 맺고 서로의 진행률과 상태를 실시간으로 공유하는 환경을 만드는 것이다. 다른 길드원들이 매일 성실하게 목표를 달성하며 레벨을 올려나가는 모습을 지켜보는 것은 사용자에게 엄청난 시너지 효과와 건강한 경쟁의식을 불러일으킨다. 서로의 성취를 자극제로 삼고 퀘스트를 수행하는 듯한 연대감을 부여함으로써, 더 이상 외롭고 지루한 자기관리의 늪에 빠지지 않게 방지하는 것이. 결과적으로 본 프로젝트는 '일정 관리'라는 본연의 목적을 넘어, 일상의 고단한 과업을 게임처럼 즐겁게 수행하고 타인과 함께 성장해 나가도록 이끄는 동기부여 플랫폼을 만들고자 하는 필요성에서 출발하게 되었다.


**1.2 Goal**

본 프로젝트의 최종 목표는 사용자의 일정 관리 효율성을 높임과 동시에, 목표 달성 과정의 지루함을 없애주는 'RPG 기반의 할 일 관리 클라이언트 시스템'과 타인의 상태를 실시간으로 공유하고 경쟁할 수 있는 '길드 서버 시스템'을 구축하는 것이다. 단순한 기록 도구를 넘어 사용자가 캐릭터를 육성하는 재미를 통해 매일의 투두리스트를 자발적으로 완수하게 만드는 것이 최종 목표이다.


**1.3 Target market**
- 매일 반복되는 일상과 목표 달성에 지루함을 느끼고 새로운 동기부여가 필요한 학생 및 직장인
- 게임적 요소를 좋아하는 사용자
- 자기 주도적 성취를 원하는 사용자
- 친구나 스터디원들과 함께 자극받으며 시너지 효과를 얻고 싶은 사용자

## 2. System context diagram

<img width="611" height="291" alt="drawio" src="https://github.com/user-attachments/assets/7ec7fdfe-0fd1-4951-92d8-4e20df6ee29c" />

* **Add Quest:** 새로운 퀘스트(할 일) 등록
* **Set Custom Point:** 퀘스트 완료 시 획득할 보상 경험치(포인트) 직접 설정
* **Display Character UI:** 현재 캐릭터의 레벨 및 진화 상태 화면 표시
* **Show Progress:** 전체 퀘스트 달성률 및 도감 수집 진행도 제공
* **Save User Data:** 사용자의 진행 데이터(경험치, 도감 등)를 로컬 저장소에 저장
* **Load User Data:** 로컬에 보관된 사용자 데이터를 시스템으로 불러오기
* **Send State Update:** 경험치 획득 등 유저 상태 변화 발생 시 서버로 데이터 전송
* **Broadcast Member State:** 서버가 변경된 유저의 상태를 길드원 모두에게 실시간으로 공유

## 3. Use case list

| Use Case | Actor | Description |
| :--- | :--- | :--- |
| **1) Sign Up** | User | 새로운 닉네임과 비밀번호를 입력하여 사용자 계정을 생성한다. |
| **2) Login** | User | 기존 닉네임과 비밀번호 일치 여부를 확인하여 개인 세션을 시작하고 접속한다. |
| **3) Data Save** | System | 현재 진행 중인 모든 데이터를 로컬 JSON 파일에 기록한다. |
| **4) Data Load** | System | 로컬에 저장된 JSON 데이터를 파싱하여 이전 상태를 복구한다. |
| **5) Add Quest** | User | 제목과 날짜를 지정하여 새로운 할 일(퀘스트)을 등록한다. |
| **6) Delete Quest** | User | 목록에서 불필요한 퀘스트를 선택하여 완전히 삭제한다. |
| **7) Set Custom Points** | User | 퀘스트 난이도에 따라 획득할 보상 포인트(EXP)를 직접 설정한다. |
| **8) Complete Quest** | User | 퀘스트 완료 체크 시 설정된 포인트만큼 경험치를 즉시 가산한다. |
| **9) Cancel Quest** | User | 완료 체크 해제 시 가산되었던 경험치를 즉시 차감하여 복구한다. |
| **10) EXP Management** | System | 포인트 가산/차감에 따라 실시간으로 누적 경험치 수치를 관리한다. |
| **11) Level Up** | System | 누적 경험치가 임계치에 도달하면 캐릭터의 레벨을 상승시킨다. |
| **12) Random Evolution** | System | 레벨업 시 무작위 확률로 캐릭터의 외형을 진화시킨다. |
| **13) Check Encyclopedia** | User | 지금까지 수집하고 해금한 모든 진화 캐릭터 목록을 조회한다. |
| **14) Network Config** | User | 서버 IP 주소와 포트 번호를 입력하여 통신 환경을 설정한다. |
| **15) Create Guild** | User | 방 이름과 비밀번호를 설정하여 새로운 실시간 길드 룸을 개설한다. |
| **16) Join Guild** | User | 고유 정보를 입력하여 이미 개설된 길드 룸에 실시간으로 입장한다. |
| **17) Real-time Sync** | Server | 유저의 레벨/캐릭터 변화를 길드원 전원의 화면에 실시간 공유한다. |
| **18) Offline Roster** | Server | 유저 종료 시에도 서버에 마지막 상태를 저장하여 명단에 유지한다. |
| **19) Leave Guild** | User | 참여 중인 길드에서 퇴장하여 개인 모드로 전환한다. |
| **20) Delegate Host** | User (Host) | 방장 퇴장 시 길드 유지를 위해 관리자 권한을 타인에게 위임한다. |
| **21) Disband Guild** | User (Host) | 방장 권한으로 길드 방을 폐쇄하고 모든 접속자를 강제 퇴장시킨다. |
| **22) Error Handling** | System | 서버 연결 실패나 통신 장애 발생 시 사용자에게 오류 알림을 제공한다. |
| **23) Data Hashing** | System | 사용자 비밀번호를 해싱 처리하여 로컬에 안전하게 보관한다. |

## 4. Concept of operation

### 1) Sign Up & Login 
| Category | Description |
| :--- | :--- |
| **Purpose** | 사용자 계정 생성 및 시스템 접속 권한 확인 |
| **Approach** | 입력받은 닉네임을 확인하여 로컬에 데이터가 없으면 비밀번호를 단방향 해시 알고리즘으로 암호화하여 저장하며 신규 계정을 생성하고, 이미 존재할 경우 닉네임과 비밀번호 모두 일치할 경우에만 해당 세션으로 접속을 승인한다. |
| **Dynamics** | 초기 실행 시 나타나는 로그인 인터페이스에서 수행 |
| **Goals** | 사용자별 독립적인 데이터 관리 환경 조성 |

### 2) Data Save & Load 
| Category | Description |
| :--- | :--- |
| **Purpose** | 로컬 저장소(JSON)를 이용한 데이터 영구 보존 |
| **Approach** | 퀘스트 내역과 캐릭터 성장 데이터를 JSON 포맷으로 파일 시스템에 쓰고 읽음으로써 프로그램 종료 후에도 진행 상황을 유지한다. |
| **Dynamics** | 앱 실행 시 로드, 주요 변동 사항 발생 및 앱 종료 시 자동 저장 |
| **Goals** | 데이터 휘발 방지 및 안정적인 데이터 연속성 보장 |

### 3) Add & Delete Quest
| Category | Description |
| :--- | :--- |
| **Purpose** | 동적인 할 일 목록 관리 |
| **Approach** | 사용자의 입력값을 바탕으로 퀘스트 객체를 리스트에 추가하거나, 고유 식별자를 참조하여 선택한 퀘스트를 목록에서 완전히 제거한다. |
| **Dynamics** | 메인 UI의 추가/삭제 버튼 클릭 시 실시간 반영 |
| **Goals** | 사용자 편의에 맞춘 자유로운 일정 관리 기능 제공 |

### 4) Set Custom Points 
| Category | Description |
| :--- | :--- |
| **Purpose** | 과업 난이도에 따른 자율적 보상 설정 |
| **Approach** | 퀘스트 생성 시 사용자가 직접 보상 경험치(EXP) 수치를 입력하며, 이 값은 해당 퀘스트의 완료 보상 기준으로 저장된다. |
| **Dynamics** | 퀘스트 등록 팝업창에서 수치 입력 시 발생 |
| **Goals** | 사용자 주도의 동기 부여 및 유연한 난이도 조절 시스템 구현 |

### 5) Complete & Cancel Quest 
| Category | Description |

## 5. Problem statement

* **Problem 1:** 서버가 꺼져 있거나 잘못된 접속 정보를 입력했을 때, 소켓 예외 처리가 미흡하면 프로그램이 예기치 않게 종료될 수 있다.
* **Problem 2:** 퀘스트 완료와 취소를 단시간에 반복할 경우, 경험치 합산 로직과 UI 업데이트 순서가 어긋나면서 수치상 오류가 생길 수 있다.
* **Problem 3:** 로컬 JSON 데이터에 쓰고 읽는 도중 프로그램이 강제 종료되면 파일 구조가 파손되어 이후 데이터 복구가 불가능해질 위험이 존재한다.
* **Problem 4:** 네트워크 통신과 게임 로직 연산을 메인 UI 스레드에서 동시에 처리할 경우, 서버 응답 대기 시간 동안 화면이 멈추는 현상이 나타날 수 있다.
* **Problem 5:** 클라이언트가 비정상적으로 연결을 끊었을 때 서버에서 해당 세션을 즉시 정리하지 못해 길드원 명단에 유령 사용자가 남는 문제가 생긴다.
* **Problem 6:** 로컬 JSON 파일에 비밀번호가 평문으로 저장될 경우, 로컬 파일 노출 시 보안 문제가 발생할 수 있으며 이를 해결하기 위한 해싱 라이브러리 연동 시 기술적 충돌이 생길 수 있다.

NFRs

*  사용자가 퀘스트를 조작했을 때 경험치 바와 레벨 수치는 즉각 업데이트되어야 한다.
*  데이터의 안전한 보존을 위해 모든 상태 변화 시 로컬 JSON 파일에 실시간으로 저장되어야 한다.
*  네트워크 연결 실패나 비밀번호 불일치 등 예외 상황 발생 시 사용자에게 명확한 안내 텍스트가 즉시 제공되어야 한다.
*  서버 접속이 불가능한 환경에서도 오프라인 모드를 통해 개인의 퀘스트 관리 및 캐릭터 조회 기능은 정상 동작해야 한다.
*  비밀번호 입력 시 화면에 글자가 노출되지 않도록 마스킹 처리가 되어야 하며 인증 실패 시 명확한 사유를 사용자에게 알려야 한다.
*  비밀번호는 원본을 알아볼 수 없도록 안전한 해싱 알고리즘을 거쳐 저장되어야 한다.

## 6. Glossary

| Terms | Description |
| :--- | :--- |
| **QuestPlanner** | 본 프로젝트로 개발된 시스템의 고유 명칭. 사용자의 할 일을 게임화하여 관리를 돕는 시스템이다. |
| **EXP** | 사용자가 할 일을 완료할 때마다 얻는 경험치. 일정 수치 도달 시 레벨업 및 캐릭터 진화의 기준이 된다. |
| **환생** | 캐릭터가 만렙(Lv.50)에 도달했을 때, 레벨과 경험치를 초기화하는 대신 '환생 횟수'를 늘리고 신비한 알로 되돌아가는 시스템. |
| **방장** | 길드를 최초로 생성한 사용자. 길드를 해산하거나 다른 길드원에게 직급을 위임할 권한을 갖는다. |
| **방명록** | 소켓 연결이 끊어진(오프라인) 사용자의 데이터도 길드 목록에 표시할 수 있도록 서버 메모리에 유지하는 데이터 구조. |
| **Socket** | 클라이언트와 서버 간의 양방향 실시간 통신을 가능하게 해주는 네트워크 프로그래밍 기술. |
| **Jackson Library** | 자바 객체(Object)를 JSON 형태의 텍스트 파일로 변환하거나 그 반대로 파싱해 주는 외부 라이브러리. |
| **포인트 지정** | 퀘스트 등록 시 사용자가 직접 설정하는 보상 값. 난이도에 따라 경험치 획득량을 자율적으로 조절할 수 있다. |
| **진화** | 캐릭터가 레벨업할 때 발생하는 시스템. 내부 난수 로직을 통해 무작위로 새로운 형태로 외형이 변하는 과정. |
| **도감** | 사용자가 지금까지 발견하고 진화시킨 모든 캐릭터의 해금 상태를 기록하고 열람할 수 있는 수집 목록. |
| **Hashing** | 사용자 비밀번호를 원본을 알 수 없는 암호문으로 변환하여 저장함으로써 로컬 저장소의 보안성을 높이는 기술. |
| **Masking** | 비밀번호 입력 시 화면에 실제 문자가 아닌 별표(*) 등을 표시하여 시각적으로 정보를 보호하는 보안 기능. |
| **Broadcasting** | 특정 사용자의 상태 변화가 발생했을 때, 서버가 같은 길드 내의 모든 클라이언트에게 동시에 데이터를 전송하는 방식. |
